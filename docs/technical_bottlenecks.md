## 瓶颈 1：量子退火使用模拟退火而非真机量子退火

### 限制描述

项目的"量子赋能AI"方向通过量子退火加速RL策略搜索。当前实现使用 D-Wave Ocean SDK 的 `neal.SimulatedAnnealingSampler`（模拟退火求解器），而非真机量子退火硬件。

**根本原因**：天衍-287 是门控量子计算机（超导量子比特），不提供量子退火接口。量子退火需要专用硬件（如 D-Wave Advantage），天衍云平台目前不具备此类设备。

### 量化影响

| 维度 | 当前状态 | 理想状态 |
|------|---------|---------|
| 求解器类型 | neal 模拟退火（经典蒙特卡洛） | D-Wave 真机量子退火 |
| 量子隧道效应 | 无（依赖热涨落） | 有（量子隧道穿越势垒） |
| 求解速度 | ~1s/次（260变量） | ~20ms/次（D-Wave Advantage） |
| 解质量 | 中等（受限于模拟退火的马尔可夫链长度） | 更优（量子并行搜索） |

### 当前应对

1. **QUBO 方法论不变**：问题建模（QUBO 映射、梯度引导编码）与真机量子退火完全兼容。切换到真机只需替换求解器后端，无需修改问题定义。
2. **诚实标注**：所有文档和报告中明确标注使用"模拟退火（neal）"，不声称使用了真机量子退火。
3. **消融实验验证**：5 Seed 消融实验显示模拟退火仍带来 +6.4% 奖励提升（虽 p=0.19 不显著，Cliff's delta=0.40 中等效应）。

### 解决路径

| 阶段 | 方案 | 时间线 | 可行性 |
|------|------|--------|--------|
| 短期 | 保持 neal 模拟退火，优化退火参数（初始温度、降温系数、扫描次数） | 9/15 前 | 高 |
| 中期 | 通过天衍云门控量子计算机实现 QAOA（量子近似优化算法）求解 QUBO 问题 | 赛后 | 中 |
| 长期 | 接入 D-Wave 量子云服务（Leap平台），实现真机量子退火 | 赛后 | 低（需额外合作） |

## 瓶颈 2：QUBO 问题规模受限于内存与计算复杂度

### 限制描述

量子退火通过 QUBO 矩阵编码神经网络权重的优化问题。QUBO 矩阵规模为 O(n²)，其中 n 为编码变量数。当前采用 `head_only` 策略，仅优化 PPO 网络的决策头（action_net + value_net），共 ~260 个参数。

### 量化影响

| 优化范围 | 参数量 | QUBO 矩阵规模 | 内存需求 | 单次退火耗时 | 可行性 |
|---------|--------|--------------|---------|------------|--------|
| head_only（当前） | 260 | 1,040² | ~8 MB | ~1s | 可行 |
| 扩展头（8张量） | ~1,000 | ~4,000² | ~128 MB | ~16s | 勉强可行 |
| 全网络 | 19,488 | 19,488² | ~2.9 GB | ~1,000s | 不可行 |

**核心矛盾**：PPO 网络的 mlp_extractor（~19,228 参数）负责特征提取，是策略性能的关键，但全量优化的内存和时间成本不可接受。

### 当前应对

1. **head_only 策略**：仅优化直接控制策略输出的决策头（260 参数），在 8MB 内存、1s 耗时内完成。
2. **4bit 编码精度**：每权重用 4 bit（1 符号位 + 3 数值位）表示，在有限变量数下最大化权重覆盖。
3. **梯度引导编码**：QUBO 目标函数融入梯度信息（min g^T Δw + λ||Δw||²），使有限精度下仍能沿正确方向优化。

### 解决路径

| 方案 | 描述 | 预期收益 |
|------|------|---------|
| 分块退火 | 将全网络参数分成多个子块，逐块退火优化 | 覆盖更多参数，内存可控 |
| 量子经典混合 | 先用经典梯度下降优化 mlp_extractor，再用退火微调决策头 | 当前已实现，可进一步优化分配策略 |
| 压缩编码 | 使用更高效的编码方案（如 Huffman 编码、分组编码）减少变量数 | 同等精度下变量数减半 |
| 网络剪枝 | 先剪枝 PPO 网络的冗余连接，减少需优化的参数量 | 参数量可减少 50-80% |

## 瓶颈 3：真机仅支持单量子比特门电路

### 限制描述

天衍云平台免费机时包对量子比特数有严格限制。实测发现，天衍-176 真机上多量子比特电路和两比特门（CNOT/CZ）频繁返回"运行失败"。为保证实验成功率，当前真机任务仅使用单量子比特门电路（如 `H Q1\nM Q1`），`FREE_TIER_MAX_QUBITS = 1`。

### 量化影响

| 电路类型 | 量子比特数 | 成功率 | 典型电路 |
|---------|----------|--------|---------|
| 单比特门 | 1 | 100%（15/15） | `H Q1\nM Q1` |
| 两比特门 | 2 | ~30%（测试值） | `H Q1\nCNOT Q1 Q2\nM Q1 Q2` |
| 多比特电路 | ≥3 | <10% | Bell 态、GHZ 态 |

**影响**：真机实验仅验证了调度系统与天衍云 API 的闭环集成（任务提交→结果查询→奖励反馈），未验证复杂量子算法（如 VQE、QAOA）的调度效果。

### 当前应对

1. **闭环验证充分**：15/15 真机任务全部成功（mock=false, real_tasks_completed=1），证明调度系统与天衍云 API 的集成是可靠的。
2. **模拟环境补充**：复杂量子算法的调度效果通过仿真环境（泊松到达、保真度模拟）验证，N=250 统计显著。
3. **配置可调**：`FREE_TIER_MAX_QUBITS` 和 `two_qubit_gates` 参数可在获得更高级别机时包后调整。

### 解决路径

| 方案 | 描述 | 前提条件 |
|------|------|---------|
| 申请更高级机时包 | 向天衍云平台申请更多量子比特权限 | 赛后与平台方沟通 |
| 逐步验证两比特门 | 先验证 2-qubit Bell 态，再扩展到 3-qubit | 获取稳定的多比特权限 |
| 仿真→真机对照 | 在仿真环境中模拟复杂量子算法，与真机结果对照 | 不依赖额外权限 |

## 瓶颈 4：量子退火训练加速的统计显著性不足

### 限制描述

5 Seed 消融实验（A: 无退火 vs B: head_only 退火）显示退火带来 +6.4% 奖励提升，但 Wilcoxon 秩和检验 p=0.190 > 0.05，统计上不显著。效应量 Cliff's delta=0.40 属于中等效应，说明有实际提升趋势，但样本量不足导致检验功效不够。

### 量化影响

| 指标 | 数值 | 判定 |
|------|------|------|
| 奖励提升 | +6.4%（50k步） | 正向趋势 |
| 最大提升 | +8.4%（35k步） | 训练中期效果明显 |
| p 值 | 0.190 | 不显著（>0.05） |
| Cliff's delta | 0.40 | 中等效应 |
| 训练时间开销 | +74.5% | 较大 |
| 波动性 | 20k-40k步区间标准差显著增大 | 退火参数需优化 |

### 当前应对

1. **诚实披露**：所有报告中明确标注 p=0.19，不声称退火效果"显著"。
2. **效应量补充**：使用 Cliff's delta 量化效应大小，弥补 p 值不显著的不足。
3. **工程定位**：将退火定位为"可选的训练加速手段"，而非核心创新点。核心创新在 AI→量子方向（RL调度）。

### 解决路径

| 方案 | 描述 | 预期效果 |
|------|------|---------|
| 增加样本量 | 从 5 Seed 扩展到 10-20 Seed | 提高检验功效，p 值可能降至 <0.05 |
| 参数调优 | 搜索退火间隔（5k→2k/8k）、退火比特数（16→8/32） | 减少波动性，提高稳定性 |
| 分阶段分析 | 在 35k 步（最大提升点）单独做显著性检验 | 可能达到显著 |
| 贝叶斯方法 | 使用贝叶斯因子替代 p 值，更适合作决策 | 避免小样本下 p 值的局限性 |

## 瓶颈 5：真机任务排队延迟不可控

### 限制描述

天衍云平台真机任务提交后需排队等待执行。实测中部分任务排队时间超过 120 秒，已将 `TASK_TIMEOUT_SECONDS` 从 120s 调整为 180s。但高峰时段排队时间仍可能超过 180s，导致任务超时失败。

### 量化影响

| 场景 | 排队时间 | 状态 |
|------|---------|------|
| 空闲时段 | <30s | 正常完成 |
| 正常时段 | 30-120s | 正常完成 |
| 高峰时段 | 120-180s | 勉强完成 |
| 极端高峰 | >180s | 超时失败 |

**影响**：在 RL 训练闭环中，真机任务超时会导致奖励信号延迟或缺失，影响训练稳定性。当前通过 `REAL_MACHINE_MAX_POLL_STEPS` 限制轮询步数，超时后降级为仿真奖励。

### 当前应对

1. **180s 超时**：`TASK_TIMEOUT_SECONDS = 180`，覆盖大部分排队场景。
2. **非阻塞提交**：`submit_to_real_machine()` 异步提交，不阻塞 RL 训练。
3. **降级机制**：`REAL_MACHINE_DEGRADE_FAIL_THRESHOLD` 连续失败超过阈值后自动降级为仿真模式。
4. **熔断器**：`CircuitBreaker` 三态转换（CLOSED/OPEN/HALF_OPEN），防止真机故障时持续重试。

### 解决路径

| 方案 | 描述 | 可行性 |
|------|------|--------|
| 预约机时 | 提前预约真机使用时段，避开高峰 | 中（需平台支持） |
| 批量提交 | 将多个真机任务批量提交，减少排队次数 | 高 |
| 缓存测量结果 | 对相同电路的测量结果缓存，避免重复提交 | 高 |
| 仿真优先 | 训练阶段使用仿真，仅在验证阶段使用真机 | 当前已部分实现 |

## 瓶颈 6：RL 策略在线推理延迟未充分验证

### 限制描述

PPO 模型在训练环境（200步/episode）中表现优异（+88.3%），但在天衍云平台生产环境中，调度决策需要在毫秒级完成。当前未对 PPO 模型的在线推理延迟进行系统测试。

### 量化影响

| 场景 | 推理延迟要求 | 当前状态 |
|------|------------|---------|
| 单次调度决策 | <100ms | 未测试 |
| 高峰时段（20任务/小时） | 持续 <100ms | 未测试 |
| 批量调度（100+任务） | <1s 总延迟 | 未测试 |

**影响**：如果 PPO 推理延迟过高，可能需要回退到规则引擎（`HybridScheduler` 已实现此降级路径），但会损失 RL 调度的性能优势。

### 当前应对

1. **混合调度器**：`HybridScheduler` 实现规则引擎+RL 三级降级（规则→RL→兜底），RL 不可用时自动回退。
2. **模型轻量化**：PPO 网络结构为 64-64 隐藏层（~19k 参数），模型文件仅 ~77KB，理论推理延迟极低。
3. **ONNX 导出**：`src/scheduler/export.py` 支持 ONNX 格式导出，可进一步优化推理速度。

### 解决路径

| 方案 | 描述 | 预期效果 |
|------|------|---------|
| 延迟基准测试 | 在不同负载下测试 PPO 推理延迟 | 量化当前性能 |
| ONNX Runtime | 使用 ONNX Runtime 替代 PyTorch 推理 | 延迟降低 2-5x |
| 模型量化 | 将 PPO 网络从 FP32 量化为 INT8 | 延迟降低 2-3x，模型体积减 4x |
| 边缘部署 | 在天衍云边缘节点部署推理服务 | 减少网络延迟 |

## 瓶颈 7：多用户公平调度缺乏量化验证

### 限制描述

项目实现了多租户配额管理（`TenantQuotaManager`），支持租户级别的量子比特上限、并发任务数、每日限额和优先级。但当前实验仅验证了单用户场景下的调度性能，未对多用户公平性进行量化评估。

### 量化影响

| 维度 | 当前状态 | 理想状态 |
|------|---------|---------|
| 租户数 | 1（单用户） | 3-10（多租户） |
| 公平性指标 | 无 | Jain's Fairness Index ≥ 0.9 |
| 优先级效果 | 字段已实现 | 量化验证 |
| 饥饿率 | 未测量 | 0%（低优先级任务不饥饿） |

**影响**：比赛方向3明确要求"多用户资源管理与公平调度"，缺乏公平性量化是落地价值维度的扣分点。

### 当前应对

1. **配额管理完整**：`TenantQuotaManager` 支持每租户的 qubit/concurrent/daily 限额和优先级（1-5级）。
2. **调度环境支持**：`env.py` 的观测空间包含 `OBS_TIME_OF_DAY`（时段信息）和 `OBS_QUANTUM_QUEUE_RATIO`（队列比率），为多用户调度提供状态信息。
3. **混合调度器**：`HybridScheduler` 的规则引擎优先处理紧急任务，可与多租户配额配合。

### 解决路径

| 方案 | 描述 | 工作量 |
|------|------|--------|
| 公平调度实验 | 3-5 租户 × 不同优先级，运行 200+ 步调度 | 1-2 天 |
| Jain's Index | 计算各租户的资源分配公平性指数 | 0.5 天 |
| 饥饿率分析 | 统计低优先级任务的最大等待时间 | 0.5 天 |
| 与 FCFS 对比 | 对比 PPO vs FCFS 在多用户场景下的公平性 | 0.5 天 |
