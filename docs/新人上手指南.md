## 第四步：搭建 Python 开发环境

### 4.1 创建虚拟环境

```bash
# 在项目根目录执行
python -m venv venv

# 激活虚拟环境
# Windows (Git Bash):
source venv/Scripts/activate
# Windows (PowerShell):
venv\Scripts\Activate.ps1
# macOS/Linux:
source venv/bin/activate
```

激活成功后，终端前会出现 `(venv)` 标识。

### 4.2 安装依赖

```bash
# 确保已激活虚拟环境（终端前有 (venv) 标识）
pip install -r requirements.txt
```

> **安装时间**：约 5-10 分钟（取决于网络速度，PyTorch 和 Qiskit 较大）。
>
> **如果安装慢**：使用国内镜像源加速
> ```bash
> pip install -r requirements.txt -i https://pypi.tuna.tsinghua.edu.cn/simple
> ```

### 4.3 安装代码格式化工具

```bash
pip install black flake8 isort
```

```ini
# 天衍云平台 API 密钥（真实模式需要，Mock 模式可随意填写）

TIANYAN_API_SECRET=your_api_secret_here

# Mock 模式（开发阶段使用，无需真实平台）

# 日志级别：DEBUG / INFO / WARNING / ERROR

```

> **安全警告**：
> - `.env` 文件已在 `.gitignore` 中，不会被提交
> - **绝对不要**把 API 密钥写在代码里
> - **绝对不要**把 `.env` 文件提交到 Git

## 第九步：日常开发流程

### 9.1 每日开发循环

```bash
# ====== 1. 早上：同步最新代码 ======
git checkout main
git pull origin main          # 拉取最新 main

git checkout feature/你的分支  # 切回自己的分支
git merge main                # 合入 main 的更新（避免累积冲突）

# ====== 2. 白天：写代码 ======
# 用 TRAE / VS Code 写代码
# 定期保存并提交

# ====== 3. 提交代码 ======
git add src/scheduler/agent.py    # 只 add 你修改的文件
git commit -m "feat: 实现DQN网络架构，完成训练循环"

# ====== 4. 推送到远程 ======
git push origin feature/你的分支

# ====== 5. 创建 Pull Request ======
# 打开 GitHub 网页，创建 PR，填写模板
# 等待至少 1 人 Review 通过后合并
```

### 9.2 分支命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 新功能 | `feature/模块名` | `feature/rl-agent` |
| 修 Bug | `fix/问题简述` | `fix/qubo-dimension-mismatch` |
| 文档 | `docs/内容` | `docs/api-reference` |
| 测试 | `test/模块名` | `test/scheduler-env` |
| 重构 | `refactor/模块名` | `refactor/reward-function` |

### 9.3 Commit 规范

```
<type>: <简短描述>
```

| type | 含义 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 实现DQN智能体训练循环` |
| `fix` | 修 Bug | `fix: 修复量子退火QUBO矩阵维度不匹配` |
| `refactor` | 重构 | `refactor: 提取奖励计算为独立函数` |
| `test` | 添加测试 | `test: 添加任务解析器边界条件测试` |
| `docs` | 文档 | `docs: 更新API接口文档` |
| `perf` | 性能优化 | `perf: 优化任务队列查找复杂度` |
| `style` | 格式化 | `style: 用Black格式化代码` |
| `chore` | 构建/工具 | `chore: 更新依赖版本` |

```bash
# ✅ 正确示例
git commit -m "feat: 实现DQN智能体的Duelling网络架构"
git commit -m "fix: 修复训练时epsilon衰减溢出bug"
git commit -m "docs: 更新API接口文档"

# ❌ 错误示例
git commit -m "update code"
git commit -m "fix bug"
git commit -m "修改了一些东西"
```

### 9.4 提交 Pull Request

**创建 PR 前的检查清单：**

```bash
# 1. 代码能跑
python -m pytest tests/ -v

# 2. 代码格式化
black src/
isort src/

# 3. 同步 main 最新代码
git checkout main && git pull origin main
git checkout feature/你的分支 && git merge main
```

**创建 PR：**

1. 推送分支后，打开 GitHub 仓库页面
2. 点击 **Compare & pull request** 按钮
3. PR 标题格式：`[模块名] 简要描述`（如 `[RL智能体] 实现Duelling DQN网络`）
4. 填写 PR 模板：
   - **做了什么**：简述改动内容
   - **测试结果**：附上测试通过的截图或日志
   - **相关 Issue**：填写 `Closes #Issue编号`（如 `Closes #4`）
   - **Review 要点**：告诉 Reviewer 重点看哪些地方
5. 点击 **Create pull request**

**Review 规则：**
- 每个 PR 至少 **1 人 Review + Approve** 才能合并
- Reviewer 应在 24 小时内响应
- 有修改意见时，在 PR 中讨论，不要私下沟通

## 注意事项

### 绝对禁止

| 禁止事项 | 原因 |
|---------|------|
| 提交 `.env` 文件 | 包含 API 密钥，属于敏感信息 |
| 提交 `venv/` 目录 | 虚拟环境不应进版本控制 |
| 提交模型文件（`.pth`, `.pt`） | 文件过大，拖慢 clone 速度 |
| 提交 `__pycache__/` 目录 | 编译缓存，无意义 |
| 在代码中硬编码 API 密钥 | 安全隐患，使用 `.env` 文件 |
| 使用 `git push --force` | 会覆盖他人提交，使用 `git revert` |
| 使用 `git reset --hard` 删除他人提交 | 代码丢失风险，使用 `git revert` |

### 重要提醒

1. **每天早上先 `git pull origin main`**：同步最新代码，避免冲突累积
2. **只 add 自己修改的文件**：不要 `git add -A` 一把梭，避免提交无关文件
3. **PR 关联 Issue**：在 PR 描述中写 `Closes #编号`，合并后自动关闭 Issue
4. **commit message 用英文前缀 + 中文描述**：如 `feat: 实现DQN网络`
5. **写完代码先跑测试**：`python -m pytest tests/ -v`，确保不破坏现有功能
6. **及时回复 PR Review**：Reviewer 提出意见后，24 小时内响应

## 常见问题 FAQ

### Q1：clone 仓库报 404

**原因**：仓库是 Public 但需要 Write 权限才能推送。请接受 GitHub 邀请或联系瑞哥添加你为协作者。

**解决**：检查邮箱，接受邀请。或联系项目经理确认是否已发送邀请。

---

### Q2：push 时提示输入密码，但输入登录密码无效

**原因**：GitHub 已不支持密码认证，需要 Personal Access Token。

**解决**：
1. 去 https://github.com/settings/tokens/new 创建 Token
2. 勾选 `repo` 权限
3. 生成后复制 Token
4. push 时密码处粘贴 Token

> Windows 用户如果用 Git Credential Manager，会弹出浏览器授权，无需手动输入 Token。

---

### Q3：commit 被拒绝，提示格式不符合规范

**原因**：PR Review 中会被指出。请使用 `feat:` / `fix:` / `docs:` 等标准前缀。

**解决**：使用正确的格式重新提交：
```bash
git commit --amend -m "feat: 修正后的提交信息"
```

---

### Q4：push main 分支被拦截

**原因**：GitHub 分支保护阻止了直接推送。这是正常的！

**解决**：切换到功能分支再推送：
```bash
git checkout feature/你的分支
git push origin feature/你的分支
# 然后去 GitHub 创建 PR
```

---

### Q5：merge main 时出现冲突

**原因**：你的分支和 main 分支修改了同一文件的同一区域。

**解决**：
```bash
git checkout feature/你的分支
git merge main

# Git 会提示冲突文件，打开文件手动编辑
# 冲突标记格式：
# <<<<<<< HEAD
# 你的代码
# =======
# main 分支的代码
# >>>>>>> main

# 编辑后保存，删除冲突标记
git add .
git commit -m "merge: 解决与main的冲突"
git push origin feature/你的分支
```

---

### Q6：pip install 安装很慢或失败

**解决**：使用国内镜像源：
```bash
pip install -r requirements.txt -i https://pypi.tuna.tsinghua.edu.cn/simple
```

PyTorch 安装慢可使用官方源：
```bash
pip install torch --index-url https://download.pytorch.org/whl/cpu
```

---

### Q7：不小心提交了敏感信息（如 API Key）

**立即操作**：
1. 不要继续 push
2. 联系项目经理
3. 撤销最近提交（保留修改）：
```bash
git reset --soft HEAD~1
# 删除敏感信息后重新提交
```

如果已经 push，需要清理 Git 历史，联系项目经理处理。

---

### Q8：日常开发禁止使用哪些命令？

以下命令在日常开发中**绝对禁止使用**：

- `git push --force` — 会覆盖他人提交，应使用 `git revert`
- `git reset --hard` — 可能导致代码丢失，应使用 `git revert`
- `git push origin main` — main 分支受保护，必须走 PR 流程

项目经理会定期检查提交记录，违规操作后果严重。

```bash
# 1. 新电脑上安装 Git 和 Python
# 2. clone 仓库

# 5. 创建分支（如果之前的分支还在远程）

```
